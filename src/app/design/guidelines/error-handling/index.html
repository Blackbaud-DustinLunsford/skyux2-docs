Guidance on how to present error messages depending on the context.

 Style and tone

Error and warning message wording should be short and succinct and use a conversational, human-voice approach. When possible, try to not only explain why the error/warning is occurring, but offer a solution. Use the Error component to provide a consistent user experience with error messaging.

Errors can frustrate a user and sometimes indicate a serious problem, so be thoughtful and helpful with the wording. Limit the use of please/sorry, be careful how you use can't/don't (try to avoid you can't or you don't), and avoid humor.

Accessibility guidelines

Don't rely on a single sensory characteristic of components such as shape, size, visual location, orientation, or sound when describing an error or offering a solution for an error. Using "above" and "below" is fine when referencing content in the appropriate reading order for the language used as long as the reference isn't ambiguous.

Page errors

There are a variety of page-level errors that can happen, usually due to a system failure rather than a user action. The Error component has default content applicable to most scenarios, although the content can be customized if it needs to be more precise or prescriptive.

Page element errors
Form

See the form design guidelines for information about validation and error handling on forms.

Modal

Modals will usually have a lot of available space for displaying error messaging. In these cases you should use the Error component.
Tab

Tabs that are used as a secondary navigation will usually have views with a lot of available space for displaying error messaging. In these cases you should use the Error component.
Tile

Since tiles have limited space and a particular tile may not be important to the user's immediate task you should keep the visual weight to a minimum by excluding the image and button, instead only showing the error message .
