Form design patterns to accommodate different scenarios and workflow requirements.

 Form containers
Inline panel

An inline panel (component is not developed yet) is appropriate for simple forms that meet the following criteria:

  • The form has few fields, its input controls are not complex, and the form will not cause significant page morphing.
  • The content that surrounds the form will not distract users.
  • The task is functionally non-modal.
Modal dialog

A modal dialog is appropriate for forms that meet any of the following criteria:

  • The task is functionally modal, and users must finish it before doing anything else.
  • Displaying the form inline would lead to substantial content shifting or require users to scroll to complete the form.
Full-page modal

A full-page modal is appropriate for forms that meet the following criteria:

  • The task is functionally modal, and users must finish it before doing anything else.
  • Users usually perform the task in a large viewport, and a full-page modal can optimize the use of space.
  • The context of the task is self-contained.
  • The form displays feedback based on the user entering data or making choices, such as showing running totals or a preview of an email being constructed.
  • The form supports a complex task for a list of objects such as filling out report cards for a class.
  • The task includes subtasks that need to be completed in context.
Sectioned form

A sectioned form is appropriate for forms that meet the following criteria:

  • Users are adding or editing a complex object and can be expected to know what they are looking for. This is common for editing pieces of a larger record and for tasks such as applying filters.
  • It’s beneficial to group related but independent forms to minimize the number of form-launching buttons on the page.

Sectioned forms are not appropriate when choices on one tab can affect other tabs because changes are potentially invisible or unpredictable to users.

Wizard

A wizard is appropriate for forms that meet the following criteria:

  • The task can be divided into discrete steps.
  • The order of the steps is important.
  • User actions in earlier steps affect later steps.
  • Users must complete or review the fields in every step.
  • Users might want to save their progress so that they can leave the form and return later.
Field layout
General field layout

Fields should generally flow vertically. Wrapping fields in the "sky-form-group" class will provide the correct amount of vertical padding between fields.

The most common exception is closely-related fields (such as First Name/Last Name) which can follow a horizontal flow. When using a horizontal flow you should use a fluid grid layout to ensure that the form scales properly to smaller viewports.

You should use the following three column widths when designing a form:

  • 1/3 width (4 fluid grid columns)
  • 1/2 width (6 fluid grid columns)
  • Full width

Single-line alphanumeric inputs can use the appropriate column width above based on the expected content.

Multi-line alphanumeric inputs, checkboxes and radio buttons should generally be full width.

Subsections

For cases where you want to visually separate a small set of fields (e.g. address fields) from the preceding/following fields you can use 20px of vertical spacing and use the "sky-subsection-heading" class for the subsection header.

Tiles

If a form collects multiple facets of information and the user needs to fill out all of them, you can use tiles to group the fields of those facets.

Sub-forms

For cases where you need to use a sub-form (e.g. when adding mulitple instances of a child record), follow the form container guidelines above. It’s acceptable to launch modal forms from within a modal form if alternate workflows are not feasible.

When you use a subform it's likely that the data for that subform will not be saved until the underlying form is saved. In those cases the subform submit button should be worded as "Done" instead of "Save" to make it clear to the user that the data is not yet saved.
Field usability
Labels

All fields should be labeled, with the label normally located to the top left of the input. You shouldn't use a colon after the label text. Adding the "sky-control-label" class to the label will apply the correct styling.

Placeholder text

You can use a small amount of placeholder text to help users with data entry. The placeholder text shouldn’t be used in place of a label.

Inline help

Refer to the user assistance guidelines for information about when to use persistent or invoked inline help content.
Selection inputs

To provide a consistent experience for users when they need to select items from a list of items, follow these guidelines to determine which controls to use.

Radio buttons

Use radio buttons when you have 2-5 options and users can only select one value.

HTML <select> field

Use an HTML <select> field when you have 6-20 values and users do not need to search for items.

Checkboxes

Use checkboxes when you have 2-5 values and users can select multiple values.

SKY UX Select Field

Use a SKY UX select field (component not developed yet) when:

  • You have user-defined values that can accumulate and become too cumbersome for the other controls.
  • You have more than 5 values and users can select multiple values.
  • You have more than 20 values and users can only select one value.
  • You need to display items with titles and descriptions.
Validation and error handling
Validation timing

When a user is filling out a form, they should be notified as soon as possible to errors so they can be corrected within the flow of their task rather than being collected until they think they are finished. The preference ranking for validation timing is:

  1. As the user is typing if there is a clear check that can be made before the typing is complete (e.g. they are typing text into a numeric input).
  2. When the field loses focus.
  3. When the form is submitted.
Required fields

Required fields are denoted with a red asterisk to the right of the field label. Adding the "sky-control-label-required" class to the label will apply this styling. You also need to add the aria-required=true property to the input to make the field accessible.

The sectioned form component includes a property to display the red asterisk in the tab panel for tabs that contain required fields.

Other validation

You should use the datepicker, timepicker, email validation, and URL validation components when collecting those types of data. You can also create custom validation as needed.

Error states

input and select fields that use Angular validation are automatically styled with a red glow when in an error state.

A validation error should be accompanied by a message letting the user know how to correct the error. This message will generally be one of two types, although there will certainly be cases where a different message is needed:

  1. [Item] is required.
  2. Enter a valid [item].

The error message should be displayed below the field with the error, with a prepended red warning icon to draw attention. Adding the "sky-error-label" class to the message will apply the correct styling.

It is sometimes the case that a form is quite long and the field that is triggering an error is scrolled out of view. To help the user identify what needs to be corrected, autoscroll to the first field that is causing an error and set focus on that field.

The sectioned form component includes a property to display a danger icon in the tab panel for tabs that contain fields with validation errors.
Form-level errors

If the form content fails to load, follow the error handling guidelines for the relevant page-level element.

If the form-level actions break (such as the Save action not working), include an error message with the "sky-text-danger" class at the top left of the form action bar.
